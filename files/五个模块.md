# Plat BO 五大模块架构设计

本项目采用高度模块化的设计，将贝叶斯优化（Bayesian Optimization, BO）流程解耦为五个核心模块。每个模块职责单一、边界清晰，共同构成了一个完整的自动优化系统。

## 模块一：初始化 (Initialization)

负责系统的启动配置、任务定义和初始数据生成。

- **职责**：
  - 定义任务配置 (`TaskConfig`)、输入输出数据模型 (`TrialInput`, `TrialOutput`)。
  - 提供数据持久化 (`task_config_store`) 和验证 (`validator`) 服务。
  - 生成初始的随机样本或空间填充样本（如 Latin Hypercube），防止冷启动阶段的过拟合。
- **关键文件**：
  - `bootstrap.py`: 任务启动入口。
  - `models.py`: 核心数据结构定义。
  - `config.py`: 运行时路径配置。

## 模块二：代理模型 (Surrogate)

负责构建和更新概率模型，是 BO 的“大脑”记忆区。

- **职责**：
  - 维护优化历史数据 (`EngineState`)。
  - 拟合概率模型（如高斯过程 Gaussian Process），提供对目标函数的预测均值和不确定性估计。
  - 负责模型的超参数更新（Belief Update）。
- **关键文件**：
  - `engine.py`: 代理模型引擎，管理数据并调用采集策略。

## 模块三：采集函数 (Acquisition)

负责决策逻辑，是 BO 的“战略中心”。

- **职责**：
  - 定义采集函数（如 Expected Improvement, EI），权衡“开发”与“探索”。
  - 执行内部优化（Internal Optimization），在代理模型面上寻找采集函数的最大值点。
  - 向系统建议下一个待评估的参数点 (`suggest_next`)。
- **关键文件**：
  - `botorch_acq.py`: 基于 BoTorch 的采集函数计算与优化逻辑。

## 模块四：优化器 (Optimizer)

负责系统的运行调度和流程控制，是 BO 的“指挥官”。

- **职责**：
  - 协调各个模块的交互，执行“建议 -> 评估 -> 更新”的闭环循环。
  - 管理并发进程（Supervisor, Worker）。
  - 提供用户交互接口（UI App）和可视化数据。
- **关键文件**：
  - `supervisor.py`: 监控和调度主进程。
  - `worker.py`: 执行具体任务的工作进程。
  - `ui_app.py`: Web 可视化服务。

## 模块五：目标函数 (Objective)

负责真实的物理评估或仿真计算，是 BO 的“交互环境”。

- **职责**：
  - 接收参数点，执行昂贵的黑盒函数评估。
  - 返回带有噪声的观测值。
  - 封装具体的工程问题（如 Branin 函数、流体仿真等）。
- **关键文件**：
  - `evaluator.py`: 评估器接口。
  - `branin_program.py`: 示例目标函数。

---

## 模块间关系

1. **Initialization** 为所有模块提供基础数据结构和配置。
2. **Optimizer** 启动并维持循环。它调用 **Surrogate** 获取建议，调用 **Objective** 进行评估。
3. **Surrogate** 依赖 **Acquisition** 来计算下一个最佳点。
4. **Acquisition** 利用 **Surrogate** 提供的模型进行内部寻优。
5. **Objective** 独立运行，仅通过标准输入输出与 **Optimizer** 交互。

这种架构确保了各部分可以独立替换（例如更换代理模型算法、更换采集策略或目标函数），而不会影响系统其他部分。
